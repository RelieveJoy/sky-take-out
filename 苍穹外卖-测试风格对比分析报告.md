## 三、代码质量维度对比

### 3.1 命名规范

**学生风格**（中文 + 简单描述）：
```java
@DisplayName("测试1 - POST /admin/employee/login 员工登录成功")
void testEmployeeLogin_Success() throws Exception { ... }
```

**AI 风格**（Given-When-Then 模式）：
```java
@DisplayName("[Login] Given valid credentials, should return JWT token and employee info")
void givenValidCredentials_whenLogin_thenReturnTokenAndEmployeeInfo() throws Exception { ... }
```

> **分析**：AI风格采用 **Given-When-Then** 模式，方法名直接描述测试场景和预期结果，无需查看代码即理解测试意图。学生风格的方法名需要结合 `@DisplayName` 才能理解。

### 3.2 测试结构

**学生风格**（过程式）：
```java
@Test
void testEmployeeLogin_Success() throws Exception {
    // 准备数据
    EmployeeLoginDTO loginDTO = new EmployeeLoginDTO();
    loginDTO.setUsername("admin");
    loginDTO.setPassword("123456");
    Employee mockEmployee = Employee.builder()...build();
    when(employeeService.login(any())).thenReturn(mockEmployee);

    // 执行并验证
    String response = mockMvc.perform(post("/admin/employee/login")...)
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.code").value(1))
            .andReturn().getResponse().getContentAsString();

    System.out.println("测试1 通过：" + response);
}
```

**AI 风格**（AAA + Given-When-Then 注释，三段式结构）：
```java
@Test
@DisplayName("[Login] Given valid credentials, should return JWT token and employee info")
void givenValidCredentials_whenLogin_thenReturnTokenAndEmployeeInfo() throws Exception {
    // Given — 准备阶段
    EmployeeLoginDTO loginDTO = new EmployeeLoginDTO();
    loginDTO.setUsername("admin");
    loginDTO.setPassword("123456");
    Employee employee = Employee.builder()...build();
    when(employeeService.login(any(EmployeeLoginDTO.class))).thenReturn(employee);

    // When & Then — 执行+验证阶段
    mockMvc.perform(post(BASE_URL + "/login")
                    .contentType(MediaType.APPLICATION_JSON)
                    .content(JSON.toJSONString(loginDTO)))
            .andExpect(status().isOk())
            .andExpect(content().contentTypeCompatibleWith(MediaType.APPLICATION_JSON))
            .andExpect(jsonPath("$.code").value(1))
            .andExpect(jsonPath("$.data.id").value(10))
            .andExpect(jsonPath("$.data.userName").value("admin"))
            .andExpect(jsonPath("$.data.name").value("AdminUser"))
            .andExpect(jsonPath("$.data.token").isString())
            .andExpect(jsonPath("$.data.token").isNotEmpty());

    // Then — 行为验证阶段
    verify(employeeService, times(1)).login(any(EmployeeLoginDTO.class));
}
```

> **分析**：
> - AI风格使用 **// Given / // When & Then / // Then** 三段式注释，结构清晰
> - AI风格增加了 **verify()** 验证Service调用，确保交互正确
> - AI风格验证了更多字段（id、token类型、Content-Type），更全面
> - 学生风格使用 `System.out.println` 输出结果，测试框架已有更好的报告机制

### 3.3 Mock 验证

**学生风格** — 仅少数测试使用 verify()：
```java
verify(employeeService, times(1)).save(any(EmployeeDTO.class)); // 仅4/20用例使用
```

**AI 风格** — 大量使用 verify + ArgumentCaptor：
```java
// verify 模式
verify(employeeService, times(1)).login(any(EmployeeLoginDTO.class));
verify(dishService, never()).listWithFlavor(any(Dish.class));  // 验证未调用
verifyNoInteractions(employeeService);  // 验证零交互

// ArgumentCaptor 精确验证
ArgumentCaptor<EmployeeDTO> captor = ArgumentCaptor.forClass(EmployeeDTO.class);
verify(employeeService, times(1)).save(captor.capture());
assertThat(captured.getUsername()).isEqualTo("new_emp");
assertThat(captured.getPhone()).isEqualTo("13800001111");
```

> **分析**：AI风格使用 **ArgumentCaptor** 捕获并断言传递给Service的具体参数值，确保Controller→Service的数据传递完全正确。学生风格仅验证调用次数，未验证参数内容。

### 3.4 参数化测试

**学生风格** — 无参数化测试，每组输入写一个独立测试。

**AI 风格** — 使用 `@ParameterizedTest` + `@CsvSource` / `@ValueSource`：
```java
// 参数化测试 — 一个方法覆盖多个场景
@ParameterizedTest
@CsvSource({
        "admin, wrong_password, PasswordErrorException",
        "'', 123456, AccountNotFoundException",
        "null_value, 123456, AccountNotFoundException"
})
@DisplayName("[Login] Given invalid credentials, should throw appropriate exception")
void givenInvalidCredentials_whenLogin_thenThrowException(
        String username, String password, String expectedException) { ... }

// 边界值测试
@ParameterizedTest
@ValueSource(ints = {1, 5, 20, 100})
@DisplayName("[Page] Given different page sizes, should accept all valid values")
void givenDifferentPageSizes_whenPageQuery_thenAcceptValidValues(int pageSize) { ... }
```

> **分析**：AI风格使用参数化测试将相似场景合并为一个方法，减少重复代码，提高维护性。学生风格每个输入组合需写独立方法，导致代码膨胀。

### 3.5 文件组织

| 对比维度 | 学生风格 | AI 风格 |
|----------|----------|---------|
| 文件数 | 1 | 3 |
| 单文件行数 | ~620 行 | ~250-350 行/文件 |
| 组织原则 | 按 Controller 合并在一个文件 | 按业务领域分文件 |
| 可维护性 | 低（修改一处影响全局） | 高（修改隔离） |
| 可读性 | 中（需在文件中搜索相关测试） | 高（文件名即表达测试范围） |

### 3.6 边界与异常测试

**学生风格** — 仅1个异常测试（账号不存在）。

**AI 风格** — 多个边界/异常场景：
```java
// 边界：空名称
givenEmployeeDTOWithEmptyName_whenCreate_thenShouldStillSave()

// 边界：不同分页大小
givenDifferentPageSizes_whenPageQuery_thenAcceptValidValues(1, 5, 20, 100)

// 异常：不存在的ID
givenNonExistentId_whenGetById_thenPropagateException()

// 缓存边界：命中/未命中
givenCacheMiss_whenListDishes_thenQueryDbAndCache()
givenCacheHit_whenListDishes_thenReturnFromCache()

// 订单生命周期：接单/拒单/派送/完成/取消/再来一单
// 店铺状态：0/1 营业/打烊
```

### 3.7 断言完备性

**学生风格典型断言**（3-5个）：
```java
.andExpect(jsonPath("$.code").value(1))
.andExpect(jsonPath("$.data.total").value(2))
.andExpect(jsonPath("$.data.records.length()").value(2))
```

**AI风格典型断言**（6-10个）：
```java
.andExpect(status().isOk())
.andExpect(content().contentTypeCompatibleWith(MediaType.APPLICATION_JSON))
.andExpect(jsonPath("$.code").value(1))
.andExpect(jsonPath("$.data.id").value(10))
.andExpect(jsonPath("$.data.userName").value("admin"))
.andExpect(jsonPath("$.data.name").value("AdminUser"))
.andExpect(jsonPath("$.data.token").isString())
.andExpect(jsonPath("$.data.token").isNotEmpty())
// + verify() + ArgumentCaptor assertions
```

## 六、学生风格的长处

尽管总体评分低于AI风格，学生风格也有其优势：

1. **学习曲线友好**：单一文件 + 中文注释，适合刚刚接触测试的大二学生理解
2. **快速上手**：一个文件包含所有测试，查找方便，适合小型项目
3. **直观易懂**：`System.out.println` 输出让初学者直接看到响应内容
4. **中文 DisplayName**：对中文母语者更易理解测试目标
5. **简单直接**：不过度工程化，每个测试只有5-10行核心逻辑

## 七、AI风格的可借鉴之处

对于从学生风格向专业风格进阶，建议优先学习以下实践：

### 7.1 立即可用（难度低）
- ✅ 使用 `BASE_URL` 常量消除硬编码URL重复
- ✅ 采用 Given-When-Then 结构组织测试代码
- ✅ 增加 `verify()` 验证Service调用
- ✅ 增加 `content().contentTypeCompatibleWith()` 验证响应类型
- ✅ 使用 `@DisplayName` 描述测试意图（中英文皆可）

### 7.2 逐步引入（难度中）
- ✅ 按业务域分文件组织测试类
- ✅ 使用 `ArgumentCaptor` 验证传递给Mock的参数
- ✅ 引入 `@ParameterizedTest` 减少重复代码
- ✅ 增加边界值和异常场景测试
- ✅ 使用 `verify(service, never())` 验证未调用

### 7.3 进阶实践（难度高）
- ✅ 引入 `@CsvSource` / `@MethodSource` 进行数据驱动测试
- ✅ 使用 `@BeforeEach` / `@AfterEach` 管理测试生命周期
- ✅ 使用 AssertJ 的 `assertThat()` 进行流式断言
- ✅ 测试Redis缓存命中/未命中两种路径
- ✅ 测试订单完整生命周期（创建→支付→接单→派送→完成）

## 八、结论

### 8.1 总体结论

通过本次对比分析，可以得出以下结论：

1. **数量与质量并存**：AI风格生成了 **59个测试用例**（学生风格的3倍），同时每个测试的质量（断言数、Mock验证、边界覆盖）也显著更高。

2. **风格差异本质**：学生风格反映了"能写测试"的水平，AI风格展现了"写好测试"的工程实践。两者差距主要体现在**结构性、完备性和可维护性**三个维度。

3. **互补关系**：学生风格的简洁性适合快速原型验证，AI风格的系统性适合持续集成和回归测试。在实际项目中，建议两者结合：初期用学生风格快速验证，稳定后用AI风格重构。

### 8.2 学习建议

对于大二软件学院学生：

> 🎯 **第一阶段**：掌握基础（当前学生风格）——能编写可运行的JUnit测试  
> 🎯 **第二阶段**：改进结构——引入 Given-When-Then、按模块分文件  
> 🎯 **第三阶段**：提升质量——增加 Mock 验证、参数化测试、边界覆盖  
> 🎯 **第四阶段**：工程实践——CI/CD 集成、覆盖率门槛、测试金字塔
